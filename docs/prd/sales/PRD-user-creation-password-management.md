## Problem Statement

MFB needs to control which people can log into the CRM and what they can access. Creating a new user is not standard CRUD — it requires coordinating an entry in Supabase Auth (`auth.users`) with a profile record in the business table (`public.sales`). Doing this through regular data provider calls is impossible because Auth user creation requires a service-role key that must never be exposed to the browser.

Password management has four distinct flows (self-service reset, admin reset, forgot-password, and set-password) that each require different levels of authority and different entry points. Without centralized orchestration, these flows risk bypassing role checks or leaving orphaned auth accounts.

The feature solves this by routing all user lifecycle operations through two purpose-built Edge Functions (`users` and `updatepassword`) that run with a service-role key on the server and enforce role checks before touching Auth or the `sales` table.

---

## Goals

1. An Admin can invite a new CRM user and receive a recovery link to share, without exposing any credentials in the browser.
2. The system handles the edge case where the invited email already exists in Auth (orphan recovery) without creating duplicate records.
3. An Admin or the user themselves can trigger a password reset by email.
4. A user completing a reset can set a new password with client-side complexity validation before submission.
5. Disabling a user preserves all historical data they created (contacts, opportunities, activities, tasks).
6. Role-based access is enforced at every layer: UI, Edge Function, and RLS policy.

## User Stories

### Admin

- As an Admin, I can invite a new sales rep by entering their name, email, and role. The system creates their auth account and gives me a recovery link to share.
- As an Admin, I can update any user's name, email, role, or disabled status.
- As an Admin, I can trigger a password reset email for any user without knowing their current password.
- As an Admin, I can disable a user account to prevent login while keeping all their historical data intact.
- As an Admin, I can re-invite a previously deleted user by email and the system restores their prior sales record rather than creating a duplicate.

### Manager

- As a Manager, I can view the list of sales users and see their roles and status.
- As a Manager, I can update my own profile (name, email) but cannot change roles or invite new users. [INFERRED — see Open Questions]

### Rep (any authenticated user)

- As a Rep, I can update my own name and email.
- As a Rep, I can reset my own password from the Security settings without Admin involvement.
- As a Rep, I can use the "Forgot Password" flow on the login page to receive a recovery email if I am locked out.
- As a Rep, I can set a new password by following the recovery link, with real-time feedback on complexity requirements.

## Risk Assessment

- **Module Risk Level:** Medium (sales module risk score 38/100)
- **Phase Assignment:** Phase 2 (medium risk — requires Phase 1 completion before changes)
- **Test Coverage:** Partial
  - `src/atomic-crm/services/sales.service.test.ts` — 615 lines, 23 cases covering `salesCreate`, `salesUpdate`, `updatePassword`, `resetUserPassword`, integration lifecycle
  - Edge Functions (`supabase/functions/users/index.ts`, `supabase/functions/updatepassword/index.ts`) — no Deno unit tests
  - `admin_update_sale` and `admin_restore_sale` RPCs — no pgTAP tests
  - Set-password page (3 auth paths) — no component tests
  - Forgot-password page — no component tests
  - `auth-callback.html` redirect — no tests
  - Orphan recovery branch — no unit test coverage
  - Disabled user login prevention (ban_duration) — no integration test verifying JWT invalidation behavior

- **Security Concerns:**
  - sec-004 (low): `SITE_URL` env var is optional in CORS validation for the `users` Edge Function. If unset, origin validation may be permissive.
  - int-api-007: Direct `fetch()` to GoTrue REST API (`/auth/v1/admin/generate_link`) bypasses the SDK. Any GoTrue API change breaks this call silently.
  - `admin_update_sale` and `admin_restore_sale` are SECURITY DEFINER RPCs with no pgTAP test coverage. Privilege escalation via these RPCs is the highest-priority untested risk.
  - `set-password-page.tsx` imports `@supabase/supabase-js` directly (CORE-001 violation — data access belongs in the provider layer).
  - Auth ban (`ban_duration: "87600h"`) does not invalidate existing JWTs. Disabled users retain access until their current session expires.
  - `COALESCE` in `admin_update_sale` prevents explicitly setting `role` or `disabled` to NULL; only non-null values overwrite existing ones.
  - Orphan recovery uses `perPage: 1000` when scanning auth users (NFR-005).

## Open Questions

The following items are flagged for human review. They cannot be resolved from code inspection alone.

1. **Manager invite rights** [REQUIRES REVIEW]: The BRD asks "Should Managers be able to invite new Reps?" The code enforces Admin-only for creation. Business decision pending.

2. **User seat limit** [REQUIRES REVIEW]: No licensing or seat cap is enforced anywhere in the code. Is there a business limit on active `public.sales` users?

3. **Disabled user data reassignment** [REQUIRES REVIEW]: When a user is disabled, their contacts and opportunities remain attributed to them. No automatic reassignment logic exists. Intentional?

4. **Timezone field scope** [INFERRED]: Timezone is used by the digest Edge Function for scheduling. Whether it also affects date display in the UI is unconfirmed.

5. **Recovery link delivery** [REQUIRES REVIEW]: The recovery URL is shown in an in-app dialog for the Admin to copy manually. The BRD notes this as an open question. Is email delivery of the link required?

6. **JWT invalidation gap** [REQUIRES REVIEW]: Disabling a user bans them in Auth but does not immediately expire active JWTs. Is this gap acceptable, or should the ban trigger an immediate session revocation?

7. **pgTAP coverage for SECURITY DEFINER RPCs** [REQUIRES REVIEW]: `admin_update_sale` and `admin_restore_sale` have no pgTAP tests. These are the highest-priority security test gaps. Recommend adding pgTAP tests before any further changes to these RPCs.

8. **Orphan recovery scalability** [REQUIRES REVIEW]: The `perPage: 1000` scan is adequate for the current user count but will silently miss users if `auth.users` exceeds 1,000 rows. A paginated scan or a keyset lookup by email is the correct fix.

9. **CORE-001 violation in set-password-page** [REQUIRES REVIEW]: `set-password-page.tsx` imports Supabase directly. This should be refactored to use the `authProvider` or a dedicated service. Confirm whether this is an acceptable exception for the auth callback path.
